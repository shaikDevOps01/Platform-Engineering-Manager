Remote-First Companies

  • Public GitHub with real commit history
  • Blog posts / technical writing
  • Async communication demonstrated in PR descriptions
  • Strong English written skills
  • Overlap hours with US/EU timezone teams

Technology Roadmap
Every major topic — why it matters, depth required, and interview expectations
How to use this section: MUST = block everything else until this is solid. HIGH = build in first 3 months. MED = build in months 3–6. BONUS = add when everything else is done.
Linux MUST — Foundation

Why it matters: Every container runs on Linux. Every cloud VM runs Linux. Every production incident involves Linux internals. If you can't navigate a broken system using only the command line, you cannot do this job.

BEGINNER
File system navigation, permissions, users, processes (ps/top/kill), package managers, bash scripting, SSH
INTERMEDIATE
systemd, cgroups, namespaces, iptables basics, /proc filesystem, strace, lsof, network troubleshooting tools (ss, netstat, tcpdump)
ADVANCED
Kernel internals (scheduler, memory management), eBPF basics, perf tools, container runtime internals, seccomp/AppArmor

Interview: "How do you troubleshoot high CPU usage on a Linux server?" — They want: ps, top, perf top, strace. Not just "restart the service."

Networking MUST — Critical

Why it matters: Kubernetes networking is complex. Service meshes, CNI plugins, NetworkPolicy, Ingress — all require solid TCP/IP, DNS, and routing knowledge. 60% of production Kubernetes issues are network problems.

BEGINNER
OSI model, TCP/IP, DNS, HTTP/HTTPS, subnets, CIDR notation, ports, firewalls
INTERMEDIATE
BGP basics, VPNs, Load balancers (L4/L7), TLS/mTLS, overlay networks, VPC peering, NAT, iptables rules
ADVANCED
Kubernetes CNI internals (Calico/Cilium), eBPF networking, service mesh data plane (Envoy), network policies at scale

Interview: "What happens when a pod tries to communicate with a service in a different namespace?" — Trace it through DNS → kube-proxy → iptables → CNI.

Kubernetes MUST — Core Skill

Why it matters: Kubernetes is the operating system of modern infrastructure. 97% of DevOps/Platform job descriptions require it. CKA certification is the most ROI-efficient credential in this field.

BEGINNER
kubectl basics, Pod lifecycle, Deployments, Services (ClusterIP/NodePort/LB), Namespaces, ConfigMaps, Secrets, ResourceQuotas
INTERMEDIATE
RBAC, HPA/VPA/KEDA, PVs/PVCs, Ingress + TLS, DaemonSets, StatefulSets, Helm charts, NetworkPolicy, multi-container pods (sidecars)
ADVANCED
Custom controllers / Operators, CRDs, multi-cluster federation, cluster autoscaler, admission webhooks, OPA/Gatekeeper, etcd backup/restore

Interview: "A pod is in CrashLoopBackOff. Walk me through your troubleshooting." Correct answer: kubectl describe pod → kubectl logs → kubectl get events → check liveness probe → check resource limits.

Terraform MUST — Industry Standard

Why it matters: Terraform is requested in 91% of job descriptions. It is the lingua franca of infrastructure automation. Companies run entire cloud environments through Terraform — you need to understand state management, modules, and remote backends, not just terraform apply.

BEGINNER
HCL syntax, providers, resources, variables, outputs, state file, terraform init/plan/apply/destroy
INTERMEDIATE
Modules, remote state (S3 + DynamoDB lock), workspaces, data sources, count/for_each, tfvars, import existing resources
ADVANCED
Custom provider development, Terragrunt, Atlantis for PR automation, state manipulation, drift detection, Sentinel policy-as-code

Interview: "How do you handle Terraform state when multiple engineers work on the same environment?" — Remote state with S3 + DynamoDB locking + separate workspaces per environment.

Docker & Containers MUST

Why it matters: Table stakes. No further justification needed. The depth you need is multi-stage builds, layer optimization, security scanning, and understanding what containerd does vs Docker.

BEGINNER
Dockerfile, build/run/push, volumes, networks, docker-compose, container lifecycle
INTERMEDIATE
Multi-stage builds, image optimization (layer caching, .dockerignore), security (non-root user, read-only FS), BuildKit, container registries
ADVANCED
OCI spec, containerd internals, image signing (cosign), SBOM generation, rootless containers, runtime security
Git & GitOps MUST

Why it matters: Git is not just version control for code — in the GitOps model, Git is the single source of truth for your entire infrastructure state. ArgoCD and FluxCD watch Git repos and reconcile cluster state continuously.

BEGINNER
Basic Git commands, branching, merging, PRs, conflict resolution, .gitignore
INTERMEDIATE
Git internals (objects, refs, history), rebase vs merge, cherry-pick, tagging, conventional commits, branch protection rules
ADVANCED
GitOps with ArgoCD/Flux, Application Sets, multi-cluster GitOps, Git as infra source of truth, automated PR creation from pipelines
CI/CD MUST

Why it matters: CI/CD is not just "write a YAML pipeline." Senior engineers design pipelines that are fast, secure, testable, and handle rollbacks gracefully. GitHub Actions is now dominant (88%) but Jenkins is everywhere in legacy environments.

BEGINNER
Basic GitHub Actions workflows, jobs/steps, triggers, secrets management, run tests + build Docker image
INTERMEDIATE
Reusable workflows, matrix strategies, caching, artifact storage, environment protection rules, OIDC for cloud auth, pipeline-as-code
ADVANCED
Self-hosted runners, custom actions, full GitOps pipeline (build → test → scan → push → ArgoCD sync), progressive delivery (canary/blue-green)
Monitoring & Observability HIGH Priority

Why it matters: Observability is what separates reactive firefighters from engineers who prevent fires. The three pillars (metrics, logs, traces) are all expected at interview level. OpenTelemetry is quickly becoming the unified standard.

BEGINNER
Prometheus metrics (gauge/counter/histogram), Grafana dashboards, log aggregation with Loki, basic alerting rules
INTERMEDIATE
SLI/SLO/SLA definitions, error budgets, distributed tracing with Jaeger/Tempo, OpenTelemetry instrumentation, custom dashboards, PagerDuty integration
ADVANCED
Correlation across all three pillars, eBPF-based observability, AI-assisted anomaly detection, observability-as-code, cardinality optimization
Python HIGH Priority

Why it matters: Python is the automation language of platform engineering. boto3 for AWS, Kubernetes Python client, custom tooling, data processing scripts for FinOps, and increasingly — AI integration. You need to be comfortable, not expert.

BEGINNER
Data structures, functions, file I/O, CLI scripts with argparse, requests library, JSON/YAML parsing
INTERMEDIATE
boto3 (AWS), kubernetes-client, asyncio basics, REST APIs (FastAPI), error handling, unit testing with pytest, packaging
ADVANCED

Go (Golang) MEDIUM — Growing Fast

Why it matters: Kubernetes, Docker, Terraform, Prometheus, ArgoCD, Flux — all written in Go. If you want to contribute to these projects, build custom operators, or write high-performance automation tools, Go is essential. Currently in 45% of senior job descriptions.

BEGINNER
Syntax basics, structs, interfaces, goroutines/channels basics, error handling, CLI tools with cobra
INTERMEDIATE
HTTP servers, context cancellation, testing, concurrency patterns, Kubernetes client-go library
ADVANCED
Kubernetes controller-runtime, kubebuilder, admission webhooks, custom resource reconcilers
DevSecOps & Security HIGH — Differentiator

Why it matters: "Shift left security" is no longer a buzzword — it is a mandate. Companies that did not embed security into their pipelines in 2022–2023 are paying for it now with breaches. This is a genuine career differentiator in 2025–2026.

BEGINNER
Container image scanning (Trivy), secret scanning (git-secrets, gitleaks), SAST basics, dependency scanning
INTERMEDIATE
OPA/Gatekeeper policies, Vault secrets management, RBAC hardening, network policies, mTLS, SBOM generation, cosign image signing
ADVANCED
Falco runtime security, eBPF-based threat detection, zero-trust architecture, supply chain security (SLSA), CIS benchmarks, Kubernetes hardening (CIS)
Service Mesh MEDIUM

Why it matters: As microservices grow, service meshes handle mTLS between services, traffic management, circuit breaking, and observability automatically. Istio is most common; Linkerd is simpler; Cilium Service Mesh is the eBPF-native future.

BEGINNER
What a service mesh solves, sidecar proxy pattern, VirtualServices, DestinationRules
INTERMEDIATE
mTLS enforcement, traffic splitting (canary), circuit breaking, Kiali for visualization, JWT authentication
ADVANCED
Multi-cluster mesh, ambient mesh (Istio), eBPF service mesh (Cilium), performance tuning, control plane HA
AI Infrastructure HIGH — Future-Proofing

Why it matters: AI infrastructure is the fastest-growing subdiscipline in platform engineering. Companies training or serving LLMs need engineers who understand GPU scheduling, high-bandwidth networking (InfiniBand/NVLink), and inference optimization. This skill pays a 30–50% premium today and will be table stakes by 2027.

BEGINNER
Kubernetes device plugins for GPU, NVIDIA device plugin, resource limits for GPU pods, basic CUDA concepts
INTERMEDIATE
GPU sharing (MIG, time-slicing), vLLM/TGI deployment, Ray clusters on Kubernetes, ML pipeline management (Kubeflow/MLflow), model registry
ADVANCED
Multi-node distributed training (NCCL), InfiniBand networking, KServe/Triton Inference Server, GPU observability, cost optimization for GPU workloads

Interview: "How would you deploy a LLM inference service that needs to scale based on request queue length?" — KEDA + vLLM + Kubernetes + Prometheus metrics.

Helm HIGH

BEGINNER
helm install/upgrade/rollback, values.yaml, chart structure, ArtifactHub
INTERMEDIATE
Writing custom charts, template functions (toYaml, indent), subcharts, library charts, chart testing
ADVANCED
Helm operator pattern, chart registry (OCI), umbrella charts, GitOps-native Helm with ArgoCD Apps

DevOps Roadmap & Comparison
How the DevOps path differs from Platform Engineering — overlap, gaps, and when to pivot

Platform Engineer

Builds systems FOR developers. Focuses on developer experience, IDP, golden paths, self-service infrastructure.

  • Kubernetes operators and CRDs
  • Internal Developer Platforms (Backstage)
  • Golden path templates
  • Platform API design
  • Developer productivity metrics (DORA)
  • Abstractions that hide infra complexity
  • Go or Python for platform tooling

DevOps Engineer

Bridges development and operations. Focuses on CI/CD, deployment automation, and operational reliability.

  • CI/CD pipeline design and maintenance
  • Deployment strategies (blue-green, canary)
  • Environment management (dev/staging/prod)
  • Incident response and on-call
  • Configuration management (Ansible)
  • Works directly with dev teams

Linux — Top Questions

What happens when you type a command in the Linux terminal? Medium
Shell reads input → tokenizes → checks for aliases/functions/builtins → if external command: fork() creates child process → execve() replaces child with the binary → kernel loads binary into memory → executes → exit code returned to parent shell. The shell also handles piping (anonymous pipes via pipe()), I/O redirection (dup2()), and environment variable inheritance.

Follow-up: "What's the difference between fork() and exec()?" fork() creates a copy of the current process; exec() replaces the current process image with a new program.
A production server has 100% CPU. Walk me through your troubleshooting. Hard
1. top or htop — identify the PID consuming CPU
2. ps aux --sort=-%cpu | head — confirm
3. strace -p <PID> — see what syscalls it's making
4. perf top -p <PID> — find the hot function (requires perf tools)
5. cat /proc/<PID>/cmdline — identify the exact command
6. lsof -p <PID> — see open files/sockets
7. Check if it's a kernel thread (kworker, ksoftirqd) — different problem
8. Check for I/O wait — iostat -x 1 — high iowait = storage bottleneck

The key: Don't just kill the process. Understand WHY it's consuming CPU before taking action. Get thread-level data with top -H -p <PID>.
Explain Linux namespaces and how they enable containers. Hard
Linux namespaces provide isolated views of system resources. The six namespace types: PID (isolated process tree, PID 1 inside container), NET (isolated network stack, virtual ethernet pair), MNT (isolated filesystem mount points), UTS (isolated hostname), IPC (isolated IPC resources like message queues), USER (UID mapping — root inside container is unprivileged outside).

Combined with cgroups (resource limits — CPU, memory, I/O), these give you the isolation primitives that Docker/containerd build on. Containers are NOT VMs — they share the kernel.

Follow-up: "What's a container escape?" A vulnerability that breaks out of namespace isolation to gain host-level access — this is why privileged containers are dangerous and why Falco/seccomp profiles matter.
How does Linux process scheduling work? Hard
The Linux kernel uses the Completely Fair Scheduler (CFS) for normal processes. CFS maintains a red-black tree ordered by virtual runtime (vruntime) — the process with the lowest vruntime always runs next. Each process has a weight based on nice value. Higher priority (lower nice) = lower weight decay = more CPU time.

Real-time processes (SCHED_FIFO, SCHED_RR) always preempt normal processes.

In Kubernetes context: CPU requests map to cgroup cpu.shares (CFS weight). CPU limits map to cfs_quota_us/cfs_period_us — hard throttling. This is why CPU throttling (not CPU usage) is often the real metric to watch.

Kubernetes — Top Questions

A pod is in CrashLoopBackOff. Walk me through your diagnosis. Medium
Step 1: kubectl describe pod <name> — look at Events section (ImagePullBackOff? OOMKilled? Liveness probe failed?)
Step 2: kubectl logs <pod> --previous — get logs from last failed container run
Step 3: Check exit code — OOMKilled shows exitCode 137, segfault shows 139, app crash shows non-zero
Step 4: kubectl get events --sort-by=.lastTimestamp — cluster-wide events
Step 5: Check resource limits — is the container being OOM killed?
Step 6: Check liveness probe settings — is the probe timing out before the app starts? (Add initialDelaySeconds)
Step 7: Try kubectl debug -it <pod> --image=busybox --target=<container> to exec into a debugging container

Most common causes: OOMKilled (increase memory limit), liveness probe too aggressive (add initialDelaySeconds), app crash on startup (missing env var/config), wrong command/args.
Explain how Kubernetes networking works end-to-end — pod to pod, cross-node. Hard
Pod-to-pod same node: Pods share a network namespace via veth pairs connecting to a bridge (cbr0 or similar). Traffic goes: pod veth → bridge → destination pod veth. No NAT.

Pod-to-pod cross-node: Depends on CNI plugin. With Flannel (VXLAN): packet encapsulated in UDP, sent to destination node, decapsulated. With Calico (BGP): routes programmed at node level, no encapsulation overhead (better performance). With Cilium (eBPF): XDP hooks replace iptables, dramatically better performance.

Pod-to-Service: kube-proxy programs iptables/IPVS rules to DNAT ClusterIP:Port to one of the healthy backend pod IPs (load balancing). With Cilium in kube-proxy replacement mode, this is done in eBPF instead.

DNS: CoreDNS resolves <service>.<namespace>.svc.cluster.local to ClusterIP. The full FQDN is the reliable form — short names depend on ndots/searchdomain settings.
How does Kubernetes scheduling work? How would you influence where a pod runs? Hard
The scheduler filters nodes (predicates), then scores them (priorities), then binds the pod to the winning node.

Filtering phase: Removes nodes where pod can't run — insufficient resources, taints, NodeSelector mismatch, affinity rules.
Scoring phase: Ranks remaining nodes — image locality, resource balance, inter-pod affinity spread.

Ways to influence scheduling:
- nodeSelector: Must match labels (hard requirement)
- nodeAffinity: required (hard) or preferred (soft) node selection
- podAffinity/podAntiAffinity: co-locate or spread pods relative to other pods
- taints + tolerations: Node repels pods unless they tolerate the taint (good for GPU nodes, spot instances)
- topologySpreadConstraints: Spread pods evenly across zones/nodes for HA
- priorityClass: Preempt lower-priority pods when resources are scarce
Explain etcd's role in Kubernetes and what happens if etcd goes down. Hard
etcd is the distributed key-value store that IS the Kubernetes cluster state. Everything: pod specs, node status, secrets, configmaps, RBAC policies — all stored in etcd. The API server is the only component that talks to etcd directly.

If etcd goes down: API server can't serve reads or writes. kubectl commands fail. New pods can't be scheduled. However, already-running pods KEEP RUNNING — kubelet on each node maintains local state and keeps containers running. The cluster is read-only from outside, but stable.

Recovery: Restore from etcd snapshot (etcdctl snapshot restore). This is why regular etcd backups are critical — CKA exam tests this. Backup frequency should match your RPO target.

HA etcd: Production clusters run 3 or 5 etcd members (Raft consensus). Requires quorum ((N+1)/2 members) to elect a leader. 3-node cluster tolerates 1 failure.
What is the difference between Deployment, StatefulSet, and DaemonSet? Medium
Deployment: For stateless apps. Pods are interchangeable. Rolling updates swap pods freely. Scale up/down without identity concerns. Use for: web servers, API services, microservices.

StatefulSet: For stateful apps. Pods have stable, ordered identities (pod-0, pod-1, pod-2). Ordered creation/deletion. Each pod gets its own PVC that persists across rescheduling. DNS: <pod>.<service>.<namespace>.svc.cluster.local. Use for: databases (MySQL, Cassandra), Kafka, Elasticsearch, ZooKeeper.

DaemonSet: Runs exactly one pod per node (or per node matching selector). Used for: logging agents (Fluentd), monitoring agents (Datadog, Prometheus node-exporter), CNI plugins (Cilium), security agents (Falco).

Follow-up often asked: "Can you scale a DaemonSet?" No — its count is determined by nodes. "What happens to a DaemonSet pod when a new node joins?" A new pod is automatically scheduled on it.

Terraform — Top Questions

How does Terraform state work? What are the risks of state corruption? Hard
Terraform state is a JSON file that maps your HCL configuration to real infrastructure resources. It stores: resource IDs (so Terraform knows which real resource corresponds to which config block), dependency graph, and metadata.

State corruption risks:
- Two engineers running terraform apply simultaneously = state conflict
- Network interruption mid-apply = partially applied state
- Manually editing state = corruption
- Deleting state = Terraform thinks resources don't exist, re-creates them (data loss risk for databases)

Mitigations: Remote state with S3 + DynamoDB locking (prevents concurrent applies). State backup with versioning on S3. Use terraform state commands for surgery (never edit JSON directly). Enable prevent_destroy lifecycle hook for critical resources.
What is the difference between count and for_each in Terraform? Medium
count: Creates N instances indexed by integer (0, 1, 2...). Problem: if you remove element 0 from a list, everything re-indexes — Terraform may destroy and recreate resources incorrectly.

for_each: Creates instances keyed by a string from a map or set. Keys are stable — removing one element doesn't affect others. Always prefer for_each when iterating over a collection of things that might change.

for_each example: for_each = toset(["us-east-1", "eu-west-1"]) creates resources with stable keys "us-east-1" and "eu-west-1".

Practical rule: Use count = var.create_resource ? 1 : 0 for conditional creation. Use for_each for iterating collections.
How do you import existing infrastructure into Terraform? Medium
Classic approach: Write the resource block in HCL first, then run terraform import <resource_type>.<name> <real_resource_id>. This pulls the resource into state. Then run terraform plan — the diff shows what attributes Terraform wants to manage vs what currently exists. You then update your HCL until plan shows no diff.

Terraform 1.5+ approach: Use import blocks in HCL + terraform plan -generate-config-out=generated.tf to auto-generate the resource configuration.

Common pitfall: Importing doesn't validate your config matches reality. Always run plan after import and fix all diffs before running apply.

AWS — Top Questions

Explain IAM roles, policies, and when to use each. Medium
IAM User: Long-lived credentials for a person or machine. Avoid for applications — use roles.
IAM Role: Assumed by services, EC2 instances, Lambda functions, EKS pods (via IRSA). Temporary credentials. Best practice for everything.
IAM Policy: JSON document defining permissions. Attached to users, groups, or roles. Types: Managed (AWS-managed or customer-managed, reusable) vs Inline (embedded in identity, one-to-one).

IRSA (IAM Roles for Service Accounts): Kubernetes pods assume IAM roles via OIDC federation. Best practice for EKS — no long-lived keys, role is scoped to exact namespace/service account. Setup: OIDC provider → IAM role with trust policy → K8s service account annotation → pod gets temp AWS creds via environment variables injected by the webhook.
How would you design a VPC for a production application? Hard
Multi-AZ design (3 AZs minimum for production):
- Public subnets: ALB, NAT Gateway, bastion host
- Private subnets: EKS nodes, EC2 app servers — no direct internet access
- Database subnets: RDS, ElastiCache — no internet access, no route to public subnets

Routing: Public subnets → Internet Gateway. Private subnets → NAT Gateway (in each AZ for resilience). DB subnets → no internet route.

Security layers: Security groups (stateful, per-resource) + NACLs (stateless, per-subnet, second line of defense). VPC Flow Logs to S3/CloudWatch for forensics. VPC endpoints for S3, DynamoDB, ECR, Secrets Manager (keep traffic off internet).

CIDR planning: Don't use 10.0.0.0/8 (too broad). Use /16 per VPC (65K IPs). /24 per subnet (250 usable). Plan for VPC peering — non-overlapping CIDRs are required.

What is an Internal Developer Platform? How do you measure its success? Hard

How would you design a multi-tenant Kubernetes platform for 50 product teams? Hard — System Design
Isolation model: Namespace-per-team OR cluster-per-team (depends on security requirements and team size). Namespace-per-team is cost-efficient; cluster-per-team is more secure (blast radius isolation).

Namespace approach (most common):
- Each team gets one or more namespaces (dev, staging, prod per team)
- RBAC: team members have edit in their namespaces, view cluster-wide
- ResourceQuotas and LimitRanges per namespace
- NetworkPolicy: deny-all by default, allow team's own namespaces
- OPA/Gatekeeper: enforce image registry, resource limits, required labels

Self-service provisioning: Teams open a PR to add themselves to a config repo → Terraform/ArgoCD creates namespace, RBAC, quotas automatically.

Shared services: Ingress controller (1 per cluster), cert-manager, External Secrets Operator, monitoring stack — all managed by platform team, accessible by all teams.
How do you handle secrets management in a Kubernetes environment at scale? Hard
The problem: Kubernetes Secrets are base64-encoded (not encrypted at rest by default). Storing secrets in Git is a serious vulnerability. Rotating secrets manually across 50 services is toil.

Solution architecture:
1. Source of truth: HashiCorp Vault or AWS Secrets Manager / GCP Secret Manager
2. Sync mechanism: External Secrets Operator (ESO) watches Vault/SM and creates K8s Secret objects automatically
3. Kubernetes: Secret objects exist ephemerally — ESO recreates them if deleted
4. Rotation: Rotate in Vault/SM → ESO detects change → updates K8s Secret → app reloads (via secret store CSI driver for file-based injection, or pod restart for env var injection)
5. Audit: Vault provides detailed audit log of every secret access
6. Access control: Vault policies mapped to Kubernetes ServiceAccounts via K8s auth method

Never store secrets in ConfigMaps. Never commit secrets to Git (even encrypted with SOPS/sealed-secrets is not ideal for large teams — use ESO instead).

System Design Questions

Design a CI/CD system that supports 500 developers with 1000 deployments per day. Hard
Key requirements: High throughput (1000 deploys/day = ~42/hr), fast feedback (CI under 5 min), reliability (pipeline failures don't block all teams), security gates, multi-environment promotion.

Architecture:
- CI: GitHub Actions with self-hosted runners on Kubernetes (actions-runner-controller). Runner pools per team size. Caching strategy: S3 or GitHub Actions cache for dependencies. Parallelization: matrix builds.
- Artifact registry: ECR with lifecycle policies (keep last 10 tags per service)
- CD: GitOps with ArgoCD. Each deploy = PR updating image tag in a manifests repo. ArgoCD auto-syncs to staging; production requires manual promotion or automated gate (test passing + canary metrics).
- Deployment strategy: Argo Rollouts with canary (5% → 25% → 100%) + automated Prometheus analysis templates
- Environments: dev (auto-deploy on merge), staging (auto-promote), prod (gate on SLO metrics)
- Observability: Pipeline metrics in Grafana (deploy frequency, lead time, failure rate)

Scale numbers: 1000 deploys/day across 3 envs = ~3000 ArgoCD sync operations. ArgoCD can handle this with proper sharding (Application sharding in ArgoCD v2.6+).
Design a disaster recovery system for a Kubernetes-based fintech platform with RPO 15 minutes and RTO 30 minutes. Hard
RPO 15 min: Backup must happen every 15 minutes. Data at rest must replicate continuously or with <15 min lag.
RTO 30 min: Full recovery in 30 minutes. This means automated failover — humans can't do it this fast manually.

Architecture:
- Kubernetes: Velero backs up cluster state every 15 min to S3 cross-region. Restore: velero restore create → ~10 min
- Database (RDS): Multi-AZ (synchronous replication, automatic failover ~90 sec) + cross-region read replica promoted to primary on DR
- Object storage (S3): Cross-region replication, continuous
- DNS failover: Route53 health checks + failover routing to DR region. TTL = 60 seconds
- Application: GitOps — ArgoCD in DR region already has all manifests. Just point at restored state
- DR runbook: Automated via Lambda/Step Functions triggered by health check failure
- Testing: Quarterly DR drill — actual failover, measure RTO/RPO achieved

The hard part: RTO 30 min is tight for database restoration. The right answer is active-passive with pre-warmed DR environment — DB already running in DR region as a read replica, promoted immediately on failover.

8. Cost optimization: Spot GPU instances (handle preemption gracefully), GPU sharing via MIG or time-slicing for smaller models

Interviewer follow-up: "How do you handle cold starts?" Pre-warm pods with minReplicas > 0. Use KV cache persistence across requests.
What is the difference between model training infrastructure and model inference infrastructure? Medium
Training infrastructure:
- Multi-node, multi-GPU — distributed training across hundreds of GPUs
- High-bandwidth GPU interconnect (NVLink, InfiniBand) critical
- Batch workload — long-running jobs (hours to weeks)
- Frameworks: PyTorch distributed, Ray Train, Horovod
- Storage: Fast shared storage (GPFS, Lustre) for training data
- Managed by: Kubernetes (KubeFlow, Ray on K8s) or specialized platforms (AWS SageMaker)

Inference infrastructure:
- Online serving — low latency, high throughput, request-response pattern
- Single GPU or small multi-GPU per replica
- Autoscaling is critical (traffic patterns)
- Frameworks: vLLM, TGI (Text Generation Inference), NVIDIA Triton, TorchServe
- Key metrics: P99 latency, tokens/second, time-to-first-token
- Cost: Much more efficient per GPU than training

Platform engineers primarily deal with inference infrastructure. Training is mostly owned by ML scientists with MLOps engineers.
